# 06_Session_Compaction: 对话记忆管理与 Token 熔断引擎

## 一、工程思想：对话记忆的"主动外科手术"

一个长时运行的 Agent 最终将面临上下文窗口的物理极限。粗暴地截断会导致 Agent "失忆"；保留全部内容则代价极高。`claw-code` 的方案是一套精密的 **主动外科手术式压缩**。

### 1. 记忆是有价值等级的
- **近期消息**（最后 N 轮）：直接保留原文，Agent 需要这些来保持连续性。
- **历史消息**：不是删除，而是"蒸馏"为结构化摘要，包括用户意图、待办事项、引用的关键文件。
- **已压缩的摘要**：当会话被二次压缩时，**旧摘要会被合并进新摘要**，而不是丢弃——形成滚动的"记忆遗传链"。

### 2. 熔断器的三个触发维度
压缩 **不是每轮都触发**，而是在满足条件时才介入：
- 可压缩段落的消息数 > `preserve_recent_messages`
- 估算的 Token 数量 ≥ `max_estimated_tokens`

### 3. 最关键的工程细节：不切割工具调用对
这是极其重要的防护机制。OpenAI 兼容 API 规定：`ToolResult` 消息**必须**紧跟在含 `ToolUse` 的 `Assistant` 消息之后。如果压缩边界恰好落在两者之间，就会产生"孤立的工具结果"，导致 API 返回 400 错误。系统通过向后扫描边界来修复这个问题。

## 二、具体实现逻辑

### 1. 智能摘要提取器
摘要不是让 LLM 总结（那样太贵），而是**程序化地从消息中提炼结构化信息**：
```rust
// 提取近期用户请求（最多3条）
let recent_user_requests = collect_recent_role_summaries(messages, User, 3);

// 推断待办（扫描含 "todo/next/pending" 的消息）
let pending_work = infer_pending_work(messages);

// 扫描消息内容中出现的文件路径
let key_files = collect_key_files(messages); // 识别 .rs/.ts/.md 后缀
```

### 2. 续返指令的 Prompt 工程（关键）
压缩后生成的系统消息包含三段精心设计的指令文本：
```
// 第一段：声明这是恢复会话
"本会话从之前的对话延续，以下摘要覆盖了早期内容。"

// 第二段（可选）：声明近期消息已保留
"近期消息已原文保留。"

// 第三段（关键）：禁止 AI 复述摘要
"请直接从中断处继续，不要询问用户，不要复述摘要，不要说'好的，继续...'之类的话。"
```
最后这句 `COMPACT_DIRECT_RESUME_INSTRUCTION` 是对抗 LLM "确认性废话"的工程实践。

### 3. Token 估算方法（轻量启发式）
系统不调用 tokenizer（那很慢），而是用 `字符数 / 4 + 1` 来快速估算 Token 数。这是业界约定俗成的经验值（英文约 4 字符/Token）。

## 三、对 Quant-Agent 的应用建议

1. **长会话成本控制**：我们与 Hermes/DeepSeek 的对话目前没有主动的上下文管理。随着使用时间增长，费用会线性增加。我们可以引入类似机制：当 `api_server` 检测到历史消息超过 20 条时，自动提取最近的市场分析要点，并用一段结构化摘要替代早期历史。
2. **K 线快照联动**：在压缩时，我们可以特别保留"上一次 AI 给出的交易建议"和"该建议对应时刻的价格"，这是量化系统最需要的"决策记忆"，不能被随意压缩掉。
3. **防止工具调用撕裂**：我们的 `api_server` 在多轮对话中不维护消息结构，未来如果接入支持 Function Calling 的 API，必须实现同样的"工具对完整性"保护。
