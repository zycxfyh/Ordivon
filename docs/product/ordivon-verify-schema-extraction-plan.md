# Ordivon Verify — Schema Extraction Plan

Status: **PROPOSAL** | Date: 2026-05-01 | Phase: PV-12
Authority: `proposal`

## 1. Existing Implicit Schemas

Current schemas exist as implicit formats within the private Ordivon repo:

| Schema | Current Format | Location |
|--------|---------------|----------|
| Verify config | JSON (ordivon.verify.json) | Fixtures + docs |
| Trust report | JSON (--json output) | Generated by ordvon_verify.py |
| Debt ledger | JSONL | governance/verification-debt-ledger.jsonl |
| Gate manifest | JSON | governance/verification-gate-manifest.json |
| Document registry | JSONL | governance/document-registry.jsonl |

These are defined by usage, not by standalone `.schema.json` files.

## 2. Public Schema Candidates

Each schema should become a standalone JSON Schema file:

| Schema File | Draft | Priority |
|------------|-------|----------|
| `ordivon.verify.schema.json` | JSON Schema draft-07 | High — users need this to write config |
| `trust-report.schema.json` | JSON Schema draft-07 | High — CI consumers parse this |
| `verification-debt-ledger.schema.json` | JSON Schema draft-07 | Medium — adopted users |
| `gate-manifest.schema.json` | JSON Schema draft-07 | Medium — adopted users |
| `document-registry.schema.json` | JSON Schema draft-07 | Medium — adopted users |

## 3. Minimal External Schema vs Native Ordivon Schema

| Aspect | External (Public) | Ordivon Native (Private) |
|--------|------------------|------------------------|
| Debt ledger | Minimal: debt_id, status, severity, opened_at, closed_at | Full: all 16+ fields per DG-6B policy |
| Gate manifest | Minimal: gate_id, display_name, command, hardness | Full: layer, expected_result_type, purpose, protects_against |
| Document registry | Minimal: doc_id, path, type, status, authority | Full: stale_after_days, last_verified, description, tags |
| Verify config | Minimal: schema_version, mode, receipt_paths | Full: debt_ledger, gate_manifest, document_registry, checks |

**Do not force external users to mimic full Ordivon DG Pack.** Lightweight validators in ordvon_verify.py already handle minimal schemas (PV-8).

## 4. Compatibility Policy

| Rule | Detail |
|------|--------|
| `schema_version` field | Starts at "0.1". All schemas carry this field. |
| Breaking changes before public alpha | Allowed. No compatibility promise. |
| After public alpha | Breaking changes require documented migration path. |
| Unknown fields | Should be ignored, not rejected (forward-compatible). |
| Required fields | Schemas define minimum required fields. Extra fields are fine. |

## 5. Validation Strategy

| Mode | Missing schema file | Invalid required field |
|------|-------------------|----------------------|
| Advisory | WARN | WARN |
| Standard | FAIL (if configured) | FAIL |
| Strict | FAIL | FAIL |

Per PV-8 semantics: advisory mode is onboarding, standard is governed, strict is full enforcement.

## 6. Extraction Process

1. Create schema files from existing fixture formats
2. Add schema to `src/ordivon_verify/schemas/`
3. Add JSON Schema validation to validators
4. Test against bad/clean/standard fixtures
5. Document in public README

## 7. Non-Activation Clause

No schemas have been extracted to standalone files. No schema validation has been implemented in the package. This is a plan only.
