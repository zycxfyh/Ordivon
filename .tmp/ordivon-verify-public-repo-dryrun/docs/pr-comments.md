## Purpose

Example PR comments generated by Ordivon Verify. Use these as templates for CI integration or manual review.

## Example 1 — READY

```
## Ordivon Verify Report

**Status**: READY | **Mode**: standard

| Check | Status |
|-------|--------|
| receipts | ✅ PASS |
| debt | ✅ PASS |
| gates | ✅ PASS |
| docs | ✅ PASS |

All selected checks passed. No contradictions detected. No overdue debt. All gates intact.

> READY means selected checks passed; it does not authorize execution.

**Reviewer note**: Ordivon Verify confirms receipt consistency. You are still responsible for code review, business logic, and merge decision.
```

## Example 2 — BLOCKED

```
## Ordivon Verify Report

**Status**: BLOCKED | **Mode**: standard

| Check | Status |
|-------|--------|
| receipts | ❌ FAIL |
| debt | ✅ PASS |
| gates | ✅ PASS |
| docs | ✅ PASS |

### Hard Failures

- **receipts/feature-x-receipt.md:12**: Claims 'Skipped: None' but nearby text suggests gate was not run.
  - **Why it matters**: Skipped verification must be registered, not claimed as 'None'.
  - **Next action**: Receipt claims no verification was skipped, but evidence shows gate(s) were not run. Correct the receipt or run the missing checks.

> READY means selected checks passed; it does not authorize execution.

**⚠️ BLOCKED**: This PR cannot be merged until the hard failure is resolved. Fix the contradictory receipt claim.
```

## Template for Agent Use

Agents running Ordivon Verify before handoff should paste this block:

```
Ordivon Verify:
  Command: uv run python scripts/ordivon_verify.py all
  Status: [READY | BLOCKED | DEGRADED]
  Hard failures: [N or list]
  Warnings: [N or list]
  Interpretation: [what this means]
  Action authorization: Not granted by Ordivon Verify
```
